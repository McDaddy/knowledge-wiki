# KV Cache 与推理资源约束

> Sources: 小nono 知识库, 2026-03-26
> Raw: [🗂️ 小nono 知识库](../../raw/🗂️ 小nono 知识库.md)

## Overview

KV Cache 是大模型推理阶段最重要的性能优化之一。它通过缓存历史 token 的 Key 和 Value，避免自回归生成时反复重算整个前缀，从而把推理过程从“重复劳动”变成“增量计算”。代价是显存占用随上下文长度线性增长，因此推理系统常常首先受限于显存和带宽，而不是算力本身。

## 为什么需要 KV Cache

在自回归生成里，每新增一个 token，模型都需要参考此前所有上下文。如果不缓存历史 K/V，模型每步都要重新计算整个前缀，时间成本会随着上下文越来越高。KV Cache 把历史 K/V 存下来，新 token 只计算自己的 Q/K/V，再与缓存拼接，这就是典型的“用空间换时间”。

## 为什么推理经常先卡在显存

推理不像训练那样保存梯度和优化器状态，但长上下文会持续推高 KV Cache 占用。因此模型文件大小只是静态成本，真正运行时还要叠加缓存成本，这也是聊天越久显存越容易吃满的根本原因。对于在线服务，显存容量和 HBM 带宽往往比纯 FLOPS 更早成为瓶颈。

## 降低 KV Cache 成本的思路

典型手段包括 GQA、MQA 和量化。它们的共同目标都是减少需要缓存的 K/V 数量，或者减少每个缓存元素占用的字节数。换句话说，优化推理不只是换更大的卡，也是在压缩历史状态的表示成本。

## 与训练的根本区别

训练的主要痛点在计算量、梯度和优化器状态；推理则更像一条长期在线的数据通道，关心的是时延、吞吐、并发和单位请求成本。把两者混在一起讨论，往往会错误估计资源瓶颈。
