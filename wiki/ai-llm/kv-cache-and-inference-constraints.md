# KV Cache 与推理资源约束

> Sources: 小nono 知识库, 2026-03-26
> Raw: [🗂️ 小nono 知识库](../../raw/🗂️ 小nono 知识库.md)

## Overview

KV Cache 是大模型推理阶段最关键的性能优化之一。它通过缓存历史 token 的 Key 和 Value，避免自回归生成时重复计算整个前缀，从而把推理从“重复重算”改造成“增量计算”。它带来的代价是显存占用会随上下文长度持续增长，因此推理系统往往更早受限于显存和带宽，而不是纯算力。

## Q1：KV Cache 解决的到底是什么问题？

在自回归生成里，每输出一个新 token，模型都需要参考此前全部上下文。如果不缓存历史 K/V，模型每一步都要把整个前缀重新计算一遍，前缀越长，重复劳动越多。

KV Cache 的作用，就是把历史 token 对应的 Key 和 Value 保存下来。这样新 token 到来时，模型只需要计算当前 token 的 Q/K/V，再与已有缓存拼接，就能完成后续注意力计算。它本质上是在用存储历史状态的方式，换取更低的重复计算成本。

## Q2：为什么说 KV Cache 是“用空间换时间”？

因为它减少的是计算量，增加的是状态占用。没有 KV Cache，系统每一步都要为整段历史重新付出计算代价；有了 KV Cache，计算变快了，但历史状态必须一直保存在显存中，直到本轮推理结束。

因此，KV Cache 的收益和代价非常明确：收益是延迟下降、吞吐提升，代价是显存会随着上下文长度线性增长。上下文越长、batch 越大、并发越高，这种显存压力就越明显。

## Q3：为什么推理经常先卡在显存，而不是先卡在 FLOPS？

推理阶段虽然不像训练那样需要保存梯度和优化器状态，但它会长期持有 KV Cache。模型文件大小只是静态成本，真正运行时还要叠加缓存成本，而缓存又会随着对话长度和并发请求数持续扩大。

这就是为什么很多在线推理系统首先受到显存容量和 HBM 带宽限制，而不是理论算力限制。对聊天型应用来说，推理越久、上下文越长，显存越容易成为第一瓶颈。

## Q4：有哪些典型方法可以降低 KV Cache 的成本？

常见思路包括 GQA、MQA 和量化。它们的目标并不完全一样，但方向一致：减少需要缓存的 K/V 数量，或者减少每个缓存元素占用的字节数。

从系统角度看，这些优化并不是“让模型更聪明”，而是“让历史状态更便宜”。推理优化很多时候不是换一张更大的卡，而是在压缩状态表示成本。

## Q5：KV Cache 为什么会改变我们理解推理系统的方式？

它提醒我们，推理系统并不是训练系统的缩小版。训练主要痛点在计算量、梯度和优化器状态；推理更像一条长期在线的数据通道，更关心时延、吞吐、并发和单位请求成本。

因此，讨论推理资源约束时，不能只盯着模型参数量，还要看上下文长度、缓存策略和显存占用路径。否则很容易高估纯算力的重要性，低估状态管理的成本。
