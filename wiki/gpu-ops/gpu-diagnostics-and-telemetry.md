## Overview

这篇文章把 GPU 训练排障里最关键的观测信号和工具链串成一条可执行的路径。原始资料里既有硬件故障模式，也有工具说明和监控字段，因此这里不把它压缩成“几个命令怎么用”，而是尽量保留从一线观测、现场固化、底层定位到长期监控的完整层次。核心思路是：先尽快判断问题属于显存、互联、PCIe、驱动还是温度功耗，再决定该看实时状态、抓完整快照，还是回到长期趋势里做判断。

## 一线观测信号

排障最先看的不是完整日志，而是那些能快速缩小问题空间的信号。结合现有 raw，可以把一线信号归成四类：XID / SXid 事件、ECC 计数、温度与频率状态、设备是否可见。

XID 是 GPU 驱动上报的错误事件，更偏 GPU 本体、显存、PCIe、引擎与驱动层；SXid 则更偏 NVSwitch 与互联侧问题。它们的价值在于能让排障者在最短时间里判断故障大致落在哪一层，而不是立刻陷入海量日志里失焦。

ECC 计数能帮助区分“已经发生了致命硬错误”与“正在逐步退化”。温度、功耗、频率则揭示了是否存在热问题、降频或负载异常。设备可见性则是最直观的硬故障信号之一：如果 `nvidia-smi` 已经看不到卡了，问题方向就会明显偏向掉卡、驱动失联或总线层异常。

## 核心工具职责

### nvidia-smi

`nvidia-smi` 是排障第一眼总览工具。它最大的价值不在“信息最全”，而在“最快给出一张当下健康快照”。从 raw 看，它最适合回答以下问题：
- 这张卡现在是否还能被系统识别。
- 当前温度、功耗、频率和利用率是否异常。
- 不可纠正 ECC 是否已经出现。
- 当前有哪些进程在占用 GPU。

这意味着当训练作业出现异常时，`nvidia-smi` 往往应该先于更重的采集工具使用，因为它能帮助排障者在几十秒内决定是继续深入，还是立刻隔离节点。

### nvidia-bug-report.sh

`nvidia-bug-report.sh` 的定位和 `nvidia-smi` 完全不同。它不是为了给你一个实时面板，而是为了把故障现场尽量完整地冻结下来。驱动版本、内核模块状态、`dmesg`、PCIe 拓扑、NVLink 状态、XID 记录、系统环境和 GPU 进程信息都会被打包进同一份快照里。

因此它尤其适用于：
- 要向 NVIDIA 提交工单或 RMA 证据。
- 需要跨团队复盘故障。
- 当前现场可能很快变化，必须先留底。

换句话说，它更像“事故取证工具”，而不是第一眼诊断面板。

### lspci -vvv

`lspci -vvv` 主要解决的是总线和设备绑定层的问题。很多训练故障最后会体现为“卡慢了”“卡掉了”“驱动没绑上”“链路降宽了”，这时就不能只停留在 GPU 工具层，而要回到 PCIe 视角看 `LnkSta`、AER 错误、链路速率/宽度和驱动绑定状态。

对于以下场景，它尤其关键：
- 怀疑 riser、主板通道或接触问题。
- 设备还在，但吞吐明显异常。
- `nvidia-smi` 看起来还活着，但训练速度或稳定性已经不正常。

### DCGM 与长期指标体系

DCGM 的价值在于把单机排障扩展成集群级长期监控。raw 里新增的资料明确了它的结构：`nv-hostengine` 常驻采集，`dcgmi` 供人工操作，`dcgm-exporter` 对接 Prometheus，Grafana 负责可视化。它覆盖的不只是利用率和温度，还包括功耗、ECC、NVLink、XID 等 200+ 指标。

这使得 DCGM 很适合回答另一个层次的问题：
- 这张卡是不是一直在慢慢变坏，而不是今天才突然坏。
- 某个机架、某个批次、某种拓扑是不是更容易出问题。
- 训练异常究竟是偶发抖动，还是长期退化趋势的结果。

因此它不是替代 `nvidia-smi`，而是把“实时快照”扩展成“时间序列证据”。

| 工具/体系 | 更适合回答什么问题 | 典型使用方式 |
| --- | --- | --- |
| `nvidia-smi` | 这张卡此刻是否健康、是否可见、是否过热、是否已有致命 ECC | 单机即时查看 |
| `nvidia-bug-report.sh` | 现在这次故障的完整现场要怎么留底 | 故障现场打包取证 |
| DCGM | 这张卡或这一批卡是不是在持续退化 | 集群长期监控与时间序列分析 |

## 关键排障字段

ECC 相关字段是 GPU 排障里信息密度最高的一类。SBE 表示可纠正错误，DBE 表示不可纠正错误。SBE 不一定立刻影响业务，但如果持续在同一位置累积，往往意味着显存正在退化；DBE 则通常已经是需要高度警惕的硬故障信号。

raw 中补充的 AFID、Bank、Address、Syndrome、Error Location、Severity、Error Type 等字段，意义就在于把“有 ECC 错误”继续往物理位置收缩。它们能帮助排障者判断：
- 是偶发软错误，还是持续发生在同一物理位置的硬缺陷。
- 是某个地址区域在退化，还是更大范围的显存健康问题。
- 是否应该考虑 Row Remapping、页面下线，还是直接 RMA。

对于 PCIe 视角，`LnkSta` 的速率和宽度、`UESta/CESta` 是否报错、`Kernel driver in use` 是否正常，都是快速缩小问题范围的字段。它们帮助把“训练异常”翻译成“总线层正在发生什么”。

| 字段 | 用途 |
| --- | --- |
| SBE / DBE | 区分可纠正错误与不可纠正错误 |
| Bank / Address | 判断错误是否集中在固定物理区域 |
| AFID | 进一步收缩到更具体的故障位置 |
| Syndrome | 辅助识别具体位错误特征 |
| `LnkSta` / `UESta` / `CESta` | 判断 PCIe 链路状态、错误与退化情况 |

## 推荐排障顺序

结合几份 raw，一个更完整的排障顺序其实可以写得更具体一些：
1. 先看 `nvidia-smi`、XID / SXid 和设备可见性，判断是不是致命故障或掉卡。
2. 如果需要保留完整上下文，立刻运行 `nvidia-bug-report.sh` 冻结现场。
3. 若怀疑总线、链路或绑定问题，切到 `lspci -vvv` 与系统日志核实 PCIe 状态。
4. 回看 DCGM / Prometheus / Grafana 的历史数据，判断这是一次性抖动还是长期退化。
5. 对 ECC 类问题进一步结合 AFID、Bank、Address、Syndrome 等字段决定是观察、重映射、隔离还是更换。

这个顺序的价值在于：它既不让人一开始就陷入海量日志，也不会只停留在单次快照而忽略长期趋势。

## 从这篇资料里提炼出来的运维方法

如果说工具本身只是“能看什么”，那这篇资料更重要的部分其实是“应该怎么把这些工具串起来”。它隐含了一套相当成熟的 GPU 运维思路：
- 先用最快的信号做故障分层，而不是先收集一切。
- 在现场可能消失前，尽快做完整快照固化。
- 对疑似总线与链路问题，回到 PCIe 视角而不是只盯 GPU 层命令。
- 对逐步退化问题，依赖 DCGM 和时间序列监控，而不是只看当下一次输出。
- 对 ECC 问题，尽量定位到物理位置和错误类型，而不是只停留在“有报错”这一层。
