# 内核绕过、RDMA 与 Mellanox 网络栈

> Sources: Mira 技术问答整理, 2026-04-19
> Raw: [服务器运维与网络知识手册](../../raw/networking-systems/2026-04-19-server-operations-and-network-handbook.md)

## Overview

高性能网络最容易混淆的地方，不是某一个名词的定义，而是不同层级的概念经常被混在一起讨论。内核网络与用户态网络描述的是数据路径位置，RDMA 描述的是数据搬运范式，InfiniBand、RoCE 与 iWARP 描述的是网络实现方式，Mellanox ConnectX、`mlx5_core`、`mlx5_ib` 和 OFED 则位于设备、驱动与软件栈层。把这些层次拆开，才能判断训练集群和高性能存储系统里的真实瓶颈。

## Q1：内核网络和用户态网络的根本区别是什么？

内核网络的典型路径是：网卡收包，进入中断和 NAPI，沿 Linux 协议栈逐层处理，再交给 socket，最后从内核态复制到用户态。它的优势在于通用性强、生态完整、安全和治理能力现成可用，因此非常适合绝大多数业务系统。

用户态网络的核心目标，则是把高频数据路径从这套通用机制里抽出来，让应用更直接地接触队列、内存和网卡能力。无论是 DPDK、XDP、AF_XDP、RDMA 还是部分 `io_uring` 场景，方向都很一致：减少系统调用、减少上下文切换、减少数据拷贝、缩短数据路径。

但用户态网络并不是“全面优于内核网络”的替代方案。它只是把问题从通用性换成了性能控制：CPU 占用更高，开发与运维复杂度更高，很多安全和流量治理能力需要自己补。因此它只适合那些确实受限于传统协议栈成本的场景。

## Q2：RDMA 到底绕过了什么，没有绕过什么？

RDMA 最常见的误解，是把它理解成“完全绕过内核”。更准确的说法是：它绕过的是高频数据路径，而不是整个控制面。

在数据路径上，RDMA 允许网卡通过 DMA 直接读写用户态内存，从而减少系统调用、上下文切换和内存拷贝。这也是它在分布式训练、GPU 通信和高性能存储中表现突出的根本原因。

但在控制路径上，内核依然参与连接建立、QP 创建、内存注册、地址解析和资源协调。也就是说，RDMA 不是把操作系统拿掉，而是把最敏感、最频繁的数据搬运部分从传统协议栈中抽离出来。

## Q3：InfiniBand、RoCEv2 和 iWARP 分别解决什么问题？

三者都属于 RDMA 体系中的不同实现路径，但适合的网络前提和运维代价并不相同。

InfiniBand 是独立的专用高性能网络体系。它的优势是性能、时延确定性和生态成熟度都很高，因此常见于 HPC 和高强度训练环境。代价是部署成本更高，通常需要独立网络基础设施。

RoCEv2 是在标准以太网上实现 RDMA。它的吸引力在于不必完全放弃现有以太网基础设施，也能获得接近 InfiniBand 的性能。但它对无损网络、PFC、ECN、拥塞控制和流量隔离非常敏感，网络侧没有调好时，性能和稳定性都容易出问题。

iWARP 基于 TCP，天然对丢包更友好，部署前提相对宽松，但时延和性能上限通常不如前两者，因此实际场景中相对少见。

因此，真正的选择标准不是“谁最快”，而是“愿意为哪种性能模型承担哪种网络复杂度”。

## Q4：为什么生产环境里经常要把 RDMA 流量和普通流量隔离？

理论上，同一块 Mellanox ConnectX 网卡可以同时承载普通 TCP 流量和 RDMA 流量，系统里也可以同时看到以太网接口和 RDMA 设备视图。但生产环境里依然常常做物理隔离，原因不是做不到共用，而是共用会明显提高复杂度。

RoCEv2 对丢包和拥塞行为高度敏感，一旦普通业务流量和 RDMA 流量混跑，PFC 配置、带宽争用、拥塞调优和故障定位都会变难。尤其在训练集群中，这种复杂度往往会直接放大成作业级稳定性问题。

因此常见做法是把普通网卡留给管理和通用业务流量，把专用 RDMA 网卡留给训练通信或高性能存储。这本质上是一种复杂度隔离策略。

## Q5：Mellanox ConnectX、`mlx5_core`、`mlx5_ib` 和 OFED 分别在栈里扮演什么角色？

Mellanox ConnectX 是硬件设备家族，负责提供高速网络与 RDMA 能力。到了 Linux 侧，`mlx5_core` 是新代际 Mellanox 网卡的核心内核驱动，负责基础设备支持；`mlx5_ib` 则是在此基础上提供 RDMA / InfiniBand 能力；`libibverbs` 和 `librdmacm` 之类的用户态库，负责让应用能实际调用这些能力。

OFED 的意义在于，它不是一个单一驱动，而是一整套生态包，里面同时包含驱动、用户态库、诊断工具和上层协议支持。真正要关注的，往往不是“驱动模块名会不会变”，而是固件、OFED、驱动、CUDA、NCCL 和上层能力是否整链匹配。

对于 GPU 训练节点，这也是为什么安装顺序通常要从底层往上：内核、OFED、GPU Driver/CUDA、`nvidia-peermem`、NCCL。这里反映的是依赖方向，而不是某种教条式步骤。

## Q6：这些概念为什么会直接影响训练和存储系统？

因为高性能系统的瓶颈经常不在单一组件，而在整条栈是否协同。训练作业是否稳定，不只取决于 GPU，也取决于机内互联、机间网络、网卡驱动、OFED、CUDA、NCCL 和网络配置是否一起工作正常。

同样地，很多看起来像“训练慢”或“存储慢”的问题，本质上并不是算力不够，而是网络路径中某一层发生了错配、拥塞、退化或配置错误。理解内核网络、用户态网络、RDMA 和 Mellanox 栈的真正分层，能帮助排障者把问题落到正确层级，而不是只盯着某一个工具或某一个厂商名词。
