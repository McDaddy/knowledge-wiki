# 内核绕过、RDMA 与 Mellanox 网络栈

> Sources: Mira 技术问答整理, 2026-04-19
> Raw: [服务器运维与网络知识手册](../../raw/networking-systems/2026-04-19-server-operations-and-network-handbook.md)

## Overview

这篇文章围绕高性能网络里最容易被简化过度的几个概念展开：内核网络与用户态网络究竟差在哪里，RDMA 究竟绕过了什么、又保留了什么，InfiniBand / RoCE / iWARP 各自解决什么问题，以及 Mellanox ConnectX、`mlx5_core`、`mlx5_ib` 和 OFED 分别位于哪一层。因为 raw 本身已经是知识点记录，所以这里尽量保留关键机制和部署约束，而不是只留下结论口号。

## 内核网络与用户态网络的真正差别

原始资料把两类网络路径对比得很清楚。内核网络的典型流程是：网卡收包，触发硬件中断，进入 softirq / NAPI，沿着 Linux 协议栈往上处理，最后把数据交给 socket，并从内核态复制到用户态。它的优势非常明显：通用性强、生态完整、iptables/conntrack 等安全与治理能力天然可用，适合绝大多数业务系统。

但代价也同样明确：
- 系统调用会带来用户态与内核态切换。
- 数据交付可能伴随额外拷贝。
- 协议栈逐层处理和中断机制会增加时延。
- 在极端吞吐和极低延迟场景下，这些成本会被放大。

用户态网络的核心思路，就是尽量把高频数据路径从这套通用机制里移出来，让应用更直接地接触队列、内存和网卡能力。原始资料提到的 DPDK、XDP、AF_XDP、RDMA、io_uring 虽然实现方式不同，但方向是一致的：减少系统调用、减少数据拷贝、缩短路径长度，或者让 CPU 以更可控的方式处理数据包。

但这并不意味着“用户态网络一定更好”。它只是把问题从通用性换成了性能控制：
- CPU 占用通常更高，尤其是轮询模型。
- 安全与流量治理能力往往需要自行重建。
- 开发和运维复杂度明显上升。
- 不适合所有业务，只适合那些真的受限于传统协议栈成本的场景。

| 维度 | 内核网络 | 用户态网络 |
| --- | --- | --- |
| 数据路径 | 经过 Linux 协议栈 | 尽量绕开通用协议栈 |
| 系统调用 | 更频繁 | 更少，甚至接近无 |
| 数据拷贝 | 通常存在 | 倾向零拷贝 |
| 延迟特征 | 更通用但更高 | 更低但更依赖场景 |
| CPU 占用 | 按需处理 | 常见轮询，CPU 占用更高 |
| 安全与治理 | iptables/conntrack 等天然可用 | 常需自行重建 |
| 适用场景 | 普通业务系统 | 极致吞吐、低时延、专用数据面 |

## RDMA 到底绕过了什么

RDMA 最容易被误解成“完全绕过内核”。原始资料把这个误解拆得很精准：真正绕过的是高频的数据路径，而不是所有控制面工作。

数据路径上，RDMA 通过网卡 DMA 直接读写用户态应用内存，减少系统调用、上下文切换和内存拷贝，这也是它能在分布式存储、GPU 通信和高性能集群里发挥作用的根本原因。

但控制路径仍然需要内核参与。连接建立、QP 创建、内存注册、地址解析、资源协调，这些并不会凭空消失。也就是说，RDMA 不是把内核整体拿掉，而是把最敏感、最频繁的搬运路径从内核抽离。这个边界非常重要，因为它决定了调试、运维和理解问题时，不应该把 RDMA 当成“完全不同于操作系统”的黑盒，而是一个部分绕过、部分依赖的混合体系。

## InfiniBand、RoCEv2 与 iWARP 的部署差异

raw 里提到了三种典型实现方式，这部分很值得保留原有密度。

InfiniBand 是独立的专用高性能网络体系。它的优点是性能、确定性和生态在高性能计算场景里都非常成熟，但代价是部署成本和独立网络要求更高。它通常出现在对延迟、吞吐和稳定性要求最严格的训练和 HPC 环境中。

RoCEv2 则是在标准以太网上实现 RDMA 能力。它的吸引力在于：不必完全抛弃现有以太网基础设施，也能获得接近 InfiniBand 的性能。但它不是“便宜版 RDMA 就天然更简单”。恰恰相反，RoCE 对无损网络、PFC、ECN、拥塞控制和流量隔离都非常敏感。很多部署问题的根因，不是应用没调好，而是网络本身没有满足它对丢包与拥塞行为的要求。

iWARP 基于 TCP，容错性更强，对丢包不那么脆弱，但性能上限和时延表现通常不如前两者，因此常见度也相对低一些。

从运维角度看，这三者最重要的不是“谁更快”，而是“你愿意为哪种性能模型付出哪种网络复杂度”。

| 方案 | 网络基础 | 优势 | 代价 |
| --- | --- | --- | --- |
| InfiniBand | 专用 IB 网络 | 性能和确定性最好 | 独立网络、成本更高 |
| RoCEv2 | 标准以太网 | 接近 IB，容易接入现网 | 高度依赖无损网络与拥塞控制 |
| iWARP | 基于 TCP 的以太网 | 对丢包更友好 | 时延和性能上限通常更低 |

## 网卡共用与流量隔离为什么这么重要

原始资料特别强调了一点：同一块 Mellanox ConnectX 网卡理论上可以同时承载普通 TCP 流量和 RDMA 流量，系统里也可能同时存在以太网接口和 RDMA 设备视图。但在生产环境，大家经常还是选择物理隔离。

原因不是“不能共用”，而是共用的代价很高：
- RoCEv2 对丢包敏感，意味着网络侧必须严格做无损配置。
- 一旦普通业务流量和 RDMA 流量混跑，PFC 和拥塞调优会变复杂。
- 带宽争用和故障定位会变难，尤其在训练集群里影响面很大。

因此训练集群常见做法是：普通网卡走管理和业务流量，专用 RDMA 网卡走训练通信或高性能存储。这不是教条，而是复杂度管理的结果。

## Mellanox ConnectX、驱动与 OFED 在栈里的位置

raw 对 Mellanox 栈的描述非常有价值，因为它不是只说“某张卡多快”，而是把软件栈一起说清楚了。

ConnectX-4 到 ConnectX-8 虽然代际、带宽和 PCIe 规格一路上升，但在 Linux 侧主驱动层面主要统一到 `mlx5_core`。这意味着升级网卡时，很多人最先想到的“驱动模块名会不会变”其实不是主要问题，真正要关注的是：
- 固件版本是否匹配硬件代际。
- OFED 版本是否支持需要的能力。
- `mlx5_ib`、verbs 库和上层组件是否一起匹配。
- 上层的 NCCL、GPUDirect、NVMe-oF 等能力是否能顺畅接上。

OFED 的意义正是在这里。它不是单一驱动，而是一整套生态包，包含内核驱动、用户态库、诊断工具和上层协议支持。对于 GPU 训练节点，raw 给出的安装顺序“内核 -> OFED -> GPU Driver/CUDA -> nvidia-peermem -> NCCL”其实非常实用，因为它反映了依赖栈从底到上的真实关系。

| 组件 | 角色 |
| --- | --- |
| `mlx5_core` | Mellanox 新代际网卡的核心内核驱动 |
| `mlx5_ib` | RDMA / InfiniBand 能力扩展 |
| `libibverbs` / `librdmacm` | 用户态 RDMA 编程与连接管理库 |
| `ibstat` / `perftest` / `mlxconfig` | 诊断、压测与设备配置工具 |
| MLNX_OFED | 把驱动、库、工具和上层支持打包成完整发行版 |

## 对训练与存储系统的直接影响

如果只记一句话，这篇文章最值得留下来的并不是“RDMA 很快”，而是：高性能系统的瓶颈经常不在单个组件，而在整条网络与软件栈是否协同。

对于训练系统来说，这意味着：
- 机内互联、机间网络、网卡驱动、OFED、CUDA、NCCL 不是独立模块，而是一条链。
- 任何一层的版本错配、链路退化、网络拥塞或配置不当，都会在训练速度和稳定性上体现出来。
- 选择内核网络、用户态网络或 RDMA，不只是选技术名词，而是在选一整套性能模型和运维成本模型。
