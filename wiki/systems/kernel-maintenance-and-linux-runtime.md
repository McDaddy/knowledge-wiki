# 内核维护与 Linux 运行时观测

> Sources: Mira 技术问答整理, 2026-04-19
> Raw: [服务器运维与网络知识手册](../../raw/networking-systems/2026-04-19-server-operations-and-network-handbook.md)

## Overview

这篇文章把几个常被分别理解的系统运维主题放在一起：Livepatch 与 DKMS 解决什么问题，内核升级为什么依然离不开重启，IRQ Off 为何会拖垮网络和调度，以及 epoll / FD 为什么是高并发服务器的基础。它们共同指向一个核心问题：怎样在不中断业务的前提下，既维护内核，又理解运行时行为。

## Livepatch 与 DKMS 不是同一类工具

Livepatch 解决的是“现在先别重启，但要临时修补正在运行的内核”；DKMS 解决的是“等你升级或重启到新内核后，第三方模块还能不能自动重新编译出来”。前者偏运行时修复，后者偏升级兼容。把两者分清之后，很多关于“为什么热补丁不能替代升级”的误解就会自然消失。

## 内核升级为什么最终仍要重启

热补丁能缓冲风险窗口，但不能替代真正切换到新内核版本。特别是跨大版本升级时，调度器、内存管理、驱动接口等变化很难通过函数级跳转完整覆盖，因此最终仍要重启。对生产环境来说，升级前比安装动作本身更重要的是验证 DKMS 模块、保留旧内核、控制升级节奏和准备回退路径。

## IRQ Off 是系统抖动的重要信号

IRQ Off 意味着 CPU 暂时关闭中断响应，这在极短关键区内是正常机制；一旦持续时间过长，就会连锁影响网卡收包、定时器触发和进程调度，最终表现为丢包、卡顿或 hard lockup。它往往不是一个独立故障，而是某段关键路径过长、驱动中断处理过慢或内存回收出现在错误位置的外部表现。

## epoll 与 FD 的运行时意义

高并发服务器的难点不在“连接多”，而在“如何只处理真正有事件的连接”。epoll 的价值就在于把无意义遍历变成就绪通知，而 FD 则是内核抽象各种资源的统一句柄。理解这层抽象后，就更容易看清 Nginx、Redis、Go runtime 和 Java NIO 为什么会在底层汇聚到类似的机制。
