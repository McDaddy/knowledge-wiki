# 进程、内存与上下文切换

## Overview

这篇文章整理程序、进程、地址空间和上下文切换的基本关系。核心主线是：程序是静态文件，进程是被操作系统加载后获得资源并运行起来的实例；每个进程有自己的虚拟地址空间，而操作系统通过 PCB 保存和恢复 CPU 上下文，让多个进程能在同一颗 CPU 上安全地交替执行。

## 程序与进程

程序是存放在磁盘上的静态代码和数据集合，进程则是这个程序被加载进内存后正在运行的实例。同一个程序可以同时启动多个进程，因此“程序是什么”和“某个程序当前跑起来的实例是什么”必须严格区分。

## 地址空间的四个核心区域

进程视角中的地址空间通常可以抽象为代码段、数据段、堆和栈。代码段承载 CPU 实际执行的指令；数据段保存初始化的全局或静态数据；堆用于运行期动态分配；栈用于函数调用现场、局部变量和返回地址。应用感知到的是连续虚拟地址，而非真实的物理内存布局。

## CPU 状态为什么能代表“进程正在做什么”

一个进程能否恢复执行，取决于 CPU 在它被切走时的寄存器状态是否完整保存。程序计数器决定下一条指令的位置，栈指针决定当前调用栈位置，通用寄存器和标志位保存中间计算结果。因此所谓“进程上下文”，本质上就是让一个进程能够从中断点继续运行的那组最小状态集合。

## 上下文切换与 PCB

在单核 CPU 上，多进程并发本质上来自快速切换而不是同时执行。操作系统会把就绪进程放进队列，轮流分配时间片。当某个进程被切走时，其寄存器和调度状态会保存到 PCB；重新获得 CPU 时，再从 PCB 恢复。上下文切换解决了两个问题：安全性，避免下一个进程看到上一个进程的现场；正确性，保证进程能从中断点继续运行。
